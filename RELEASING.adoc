= Releasing quickfixj-spring-boot-starter

This project uses CI-friendly versioning: the version is the `<revision>` property in `pom.xml`, and the `flatten-maven-plugin` resolves it before publishing. The `README.adoc` references that version through a single AsciiDoc attribute (`:current-version:`), so each release only edits one line in each file. Releases go to Maven Central via `org.sonatype.central:central-publishing-maven-plugin`, which auto-publishes (`autoPublish=true`, `waitUntil=published`).

The `release` profile runs `maven-enforcer-plugin` with `requireReleaseVersion` and `requireReleaseDeps`, so a release build will fail fast if the version is a SNAPSHOT or any dependency is.

Two active release lines:

* `main` — Spring Boot 4.x line
* `3.x` — Spring Boot 3.x line

The flow below works for either; only the base branch and version numbers change.

== Prerequisites

* GPG key available locally. Personal signing key id: `E669C3E4ABB2AC58`.
* GPG passphrase already in `~/.m2/settings.xml` under the `ossrh` profile.
* Sonatype Central credentials in `~/.m2/settings.xml` under server id `central` (matches `publishingServerId` in the pom).
* Clean working tree on the release base branch.

== Release flow

Template — replace `<RELEASE>` with the version being cut, `<NEXT_SNAPSHOT>` with the next development version, and `<BASE>` with `main` or `3.x`.

[source,bash]
----
# 0. Sync the base branch
git checkout <BASE> && git pull origin <BASE>

# 1. Cut a release branch
git checkout -b release/<RELEASE>

# 2. Bump the revision property (single source of truth in pom.xml)
mvn versions:set-property -Dproperty=revision -DnewVersion=<RELEASE>
mvn versions:commit

# 3. Update the README version attribute (single line)
sed -i "s|^:current-version:.*|:current-version: <RELEASE>|" README.adoc

# 4. Verify the build is green and signing + enforcer rules pass
#    (release profile: gpg:sign on verify, enforcer on validate)
mvn clean verify -Prelease -Dgpg.keyname=E669C3E4ABB2AC58

# 5. Commit, tag, push branch + tag
git add -A && git commit -m "Release <RELEASE>"
git tag -s <RELEASE> -m "Release <RELEASE>"
git push origin release/<RELEASE>
git push origin <RELEASE>

# 6. Deploy to Maven Central (blocks until published)
mvn clean deploy -Prelease -Dgpg.keyname=E669C3E4ABB2AC58

# 7. Merge the release commit into the base branch
#    Option A — fast-forward (personal repo):
git checkout <BASE>
git merge --ff-only release/<RELEASE>
git push origin <BASE>
#    Option B — open a PR for review, then squash/merge.

# 8. Bump to next SNAPSHOT on the base branch
mvn versions:set-property -Dproperty=revision -DnewVersion=<NEXT_SNAPSHOT>
mvn versions:commit
git add -A && git commit -m "Prepare for next development iteration"
git push origin <BASE>

# 9. Create a GitHub release with auto-generated notes
gh release create <RELEASE> --generate-notes --title "<RELEASE>" --target <BASE>

# 10. Clean up the release branch
git branch -d release/<RELEASE>
git push origin --delete release/<RELEASE>
----

== Notes & gotchas

* `versions:set` doesn't play well with `${revision}`; `versions:set-property -Dproperty=revision` is the right tool.
* The flatten plugin (`flattenMode=resolveCiFriendliesOnly`) substitutes `${revision}` in the deployed POMs, so consumers never see the placeholder.
* The release profile's enforcer rules (`requireReleaseVersion`, `requireReleaseDeps`) run during `validate`, so even a `mvn validate -Prelease` will reject a SNAPSHOT version or SNAPSHOT dependency before any deploy work happens.
* `central-publishing-maven-plugin` has `autoPublish=true` and `waitUntil=published`. The `deploy` step blocks until Central confirms publication — don't kill it early.
* Tag-then-deploy is intentional: if deploy fails, the tag points at a reproducible commit you can retry from. If you'd rather not tag a failed release, swap steps 5 and 6.
* Maven Central does not support unpublishing — once `autoPublish` succeeds, the artifact is permanent. To rehearse safely, override the publishing plugin with `-DskipPublishing=true` for a dry run that uploads but doesn't release.
* Tags here have no `v` prefix (e.g. `4.1.0`, not `v4.1.0`) — keep that convention.
